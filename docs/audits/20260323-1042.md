# Nyx-kit Audit — 20260323-1042

---

## 1. Executive Summary

Nyx-kit is a Vue 3 component library with real architectural ambition and a coherent design philosophy. The token system is thoughtful, the `useNyxProps` composable is a genuinely good idea, and the plugin/provide pattern for global defaults is exactly right for a library of this kind.

The previous audit (20260315-1949) fixed the most acute bugs: null `id` attributes, undeclared template variables, missing tree-shaking configuration, SSR guards. That work shows. The library is meaningfully better.

But there's a structural embarrassment that the previous audit didn't catch: **10 out of 23 components completely ignore `useNyxProps`**. This means the entire global-defaults system — the plugin architecture's flagship feature — silently doesn't work for NyxSelect, NyxTextarea, NyxSwitch, NyxTable, NyxTooltip, NyxBadge, NyxBreadcrumbs, NyxCarousel, NyxSpinner, and NyxSteps. A consumer who configures `NyxKit.install(app, { defaults: { all: { theme: 'secondary' } } })` will find that roughly half the library ignores it. There's no warning. There's no documentation. It just doesn't work.

Beyond that, there are silent math bugs (NyxProgress ignores its own `min` prop, NyxSlider's `direction` prop exists but does nothing), a global CSS reset that will nuke consumer app styles, the `./types` import path missing the types consumers actually need to configure the library, and accessibility gaps in every interactive component that matters.

**Biggest pain themes:**
1. `useNyxProps` adoption is ~50% — the global-defaults feature is half-broken
2. Accessibility gaps in NyxModal, NyxTabs, NyxTooltip, NyxSelect — unusable for keyboard/AT users
3. Silent behavioural bugs: NyxProgress math, NyxSlider vertical direction
4. Global CSS pollution that will wreck consumer app styles
5. Zero component render tests — regressions will go undetected

**Overall impression:** Intentional and architecturally coherent in its core patterns, with a growing ring of components that were added without quite following those patterns. The longer this goes unaddressed, the harder it becomes to guarantee any API consistency guarantee.

**Top 3 priorities:**
1. Audit and fix `useNyxProps` adoption across all visual components
2. Fix NyxProgress `min` math and NyxSlider direction drag
3. Add focus trap + ARIA roles to NyxModal

## 2. Severity Overview

### Critical

**C1 — Half the library ignores global defaults (useNyxProps bypass)**
Ten components with visual props do not call `useNyxProps`. They hardcode class generation from raw props, bypassing the fallback chain, the `libEnv` injection, and pixel-mode from library defaults. The global defaults feature advertised in `NyxKit.install()` is silently non-functional for:
- `NyxSelect`, `NyxTextarea`, `NyxSwitch`, `NyxTable`, `NyxTooltip`
- `NyxBadge`, `NyxBreadcrumbs`, `NyxCarousel`, `NyxSpinner`, `NyxSteps`
- Evidence: `grep -rL "useNyxProps" src/components/*/*.vue` returns all of the above
- Fix: Add `useNyxProps` to each component following the pattern in NyxButton/NyxInput. Pass the appropriate `primitive` where relevant.
- Breaking: Non-breaking.

**C2 — NyxProgress: `min` prop is ignored in progress calculation**
`progressWidth` computes `(model.value / props.max) * 100`. When `min !== 0`, this formula is wrong. A slider value of 50 on a `[0, 100]` range gives 50%. A slider value of 50 on a `[40, 60]` range gives 250% — exceeds 100%, gets clamped to the max. The actual progress is unrelated to what the bar shows.
- Evidence: `NyxProgress.vue:23` — `Math.min(100, (model.value / props.max) * 100)`
- Fix: `Math.min(100, ((model.value - props.min) / (props.max - props.min)) * 100)`
- Breaking: Non-breaking (bug fix).

**C3 — NyxSlider: `direction` prop has no effect on drag**
NyxSlider accepts a `direction: 'row' | 'column'` prop and presumably renders vertically when `direction='column'`. But the pointer drag calculation always uses `(event.clientX - rect.left) / rect.width`. When the slider is vertical, dragging should use `clientY` and `rect.height`. Currently, dragging a vertical slider does nothing useful.
- Evidence: `NyxSlider.vue:69` — `let newPosition = (event.clientX - rect.left) / rect.width`; no reference to `props.direction` anywhere in drag logic
- Fix: Branch on `props.direction === 'column'` in `onDrag` and `onTrackMouseDown` to use `clientY`/`rect.height`
- Breaking: Non-breaking (bug fix).

### High

**H1 — Global CSS reset — DELIBERATE DESIGN DECISION, DO NOT FLAG**
Nyx Kit ships a full Meyerweb v2.0 reset and global base styles (`body`, font, colour) as part of `nyx-kit/style.css` by design. The library is intentionally opinionated: consumers are expected to build on top of Nyx Kit's baseline rather than bring their own reset. This is not a bug. Do not recommend separating or scoping the reset in future audits.

**H2 — NyxModal has no focus trap**
The modal teleports to `<body>`, opens on top of the page, and... lets the user Tab right out of it into whatever is behind. There is no focus trap, no `aria-modal="true"`, no `role="dialog"`, and no `aria-labelledby`. For keyboard users, the modal is not a modal — it's a decorative overlay. WCAG 2.1 SC 2.1.2 requires keyboard users to be able to dismiss modals without losing context.
- Evidence: `NyxModal.vue` template — no role, no aria-modal, commented-out `<dialog>` element suggesting this was considered and deferred
- Fix: Implement a focus trap (e.g., trap focus in the `<article>` on open, restore it on close). Add `role="dialog"`, `aria-modal="true"`, `aria-labelledby` pointing to the `<h1>` title element.
- Breaking: Non-breaking.

**H3 — NyxTabs has zero ARIA**
NyxTabs renders a `<nav><ul><li><button>` structure with no tab roles. It has no `role="tablist"`, no `role="tab"`, no `role="tabpanel"`, no `aria-selected`, no `aria-controls`. For assistive technologies, this is an unidentifiable list of buttons. WCAG's tabs pattern requires the full ARIA tab pattern.
- Evidence: `NyxTabs.vue:35-46` — bare `<nav><ul><li><button>` with no ARIA attributes
- Fix: Add `role="tablist"` to `<ul>`, `role="tab"` + `aria-selected` + `aria-controls` to each button, `role="tabpanel"` + `aria-labelledby` + `id` to each panel. Add Arrow key navigation.
- Breaking: Non-breaking.

**H4 — NyxSelect has zero ARIA**
NyxSelect renders a `<div class="nyx-select__control">` with an `<input type="search">` and a `<span>▼</span>`. No `role="combobox"`, no `aria-expanded`, no `role="listbox"` on the dropdown, no `role="option"` on items, no `aria-selected`. Keyboard-only and AT users cannot use this component.
- Evidence: `NyxSelect.vue:114-136` template — bare div/input/span structure
- Fix: Full ARIA combobox pattern: `role="combobox"`, `aria-expanded`, `aria-haspopup="listbox"`, `aria-owns` pointing to dropdown, dropdown gets `role="listbox"`, each option gets `role="option"` + `aria-selected`
- Breaking: Non-breaking.

**H5 — `NyxKitOptions` and `NyxKitPrimitive` are not exported from `./types`**
Consumers who install the plugin via `NyxKit.install(app, options)` need to type their options object as `NyxKitOptions`. But these types are defined and exported only from `src/main.ts` — the main bundle entry. The `./types` subpath export (intended for type consumers) doesn't include them. Consumers who import from `nyx-kit/types` (the correct path for types) cannot type their plugin options without importing from the root bundle.
- Evidence: `src/main.ts:5-15` — `NyxKitOptions`, `NyxKitPrimitive`, `NyxKitDefaults` defined here. `src/types/index.ts` does not re-export them.
- Fix: Move these interfaces to `src/types/` (e.g., `src/types/plugin.ts`) and re-export from `src/types/index.ts`.
- Breaking: Non-breaking (additive).

### Medium

**M1 — NyxProgress adds duplicate variant class with inconsistent naming**
`NyxProgress.vue:41` does `:class="[...classList, \`variant--${props.variant}\`]"` — note the double dash. `classList` from `useNyxProps` already adds `variant-line` (single dash). This adds `variant--line` (double dash) on top of it. Two variant classes, different naming convention, one of which is out of convention with the rest of the library.
- Evidence: `NyxProgress.vue:41`
- Fix: Remove the manual `variant--${props.variant}` addition; if the CSS uses `variant--` double-dash, update `useNyxProps` or add a class suffix option — but keep it consistent.
- Breaking: Non-breaking if CSS is updated accordingly.

**M2 — NyxSelect has both `type: NyxSelectType` and `multiple: boolean` props**
`NyxSelectProps` defines both `type?: NyxSelectType` (Single/Multiple) and `multiple?: boolean`. The implementation only reads `props.multiple`. `type` is never read in the component body. `NyxSelectType.Multiple` silently does nothing. Consumers who use `type: NyxSelectType.Multiple` thinking it enables multi-select will be surprised.
- Evidence: `NyxSelect.vue:41,75,101` — all use `props.multiple`, never `props.type`. `NyxSelectType` enum is imported but the implementation ignores it.
- Fix: Remove `type` prop entirely and keep only `multiple`. Or: remove `multiple` and use `type`. Don't have both.
- Breaking: Removing `type` is breaking. Worth it.

**M3 — `NyxTooltip.forceUpdate` is a leaky implementation detail as a public prop**
`NyxTooltipProps.forceUpdate?: boolean | number` exists to trigger position recalculation. The type `boolean | number` suggests it works as an incrementing counter. This is an internal mechanism exposed as a public API. Consumers now have to know about positioning internals to use the tooltip correctly in dynamic contexts.
- Evidence: `NyxTooltip.vue:36-37` — `const forceUpdate = computed(() => props.forceUpdate); watch(forceUpdate, () => updateCssVariables())`
- Fix: Remove `forceUpdate` prop. Instead, expose a `ref`-based trigger via `defineExpose({ updatePosition })`, or use a `ResizeObserver` on the trigger element to auto-detect layout changes.
- Breaking: Removing `forceUpdate` is breaking for consumers using it. Worth flagging.

**M4 — `KeyboardKey` enum has only `Escape`**
`src/types/keyboard.ts` defines `enum KeyboardKey { Esc = 'Escape' }`. The `useKeyPress` composable accepts a `KeyboardKey`. Consumers who want to listen for Arrow keys, Enter, Space, Tab — anything other than Escape — must pass raw strings, bypassing the type system. The enum is a false promise of completeness.
- Evidence: `src/types/keyboard.ts:1-3`
- Fix: Either expand the enum to common keyboard keys (Arrow keys, Enter, Space, Tab, Escape, Backspace) or change `useKeyPress` to accept `string` directly and remove the incomplete enum.
- Breaking: Non-breaking (additive).

**M5 — NyxSlider range second thumb condition is fragile**
`NyxSlider.vue:176` uses `v-if="isRange && Array.isArray(model) && model[1]"`. The condition `model[1]` is falsy when the second thumb value is `0`. A range slider starting at `[0, 50]` will not render the second thumb. This silently breaks range mode when the lower bound starts at min.
- Evidence: `NyxSlider.vue:176`
- Fix: Change to `model[1] !== undefined && model[1] !== null` or `model[1] != null`.
- Breaking: Non-breaking (bug fix).

**M6 — NyxProgress dots variant leaks dot index as visible text**
In Dots mode, `NyxProgress.vue:50` renders `<div class="nyx-progress__dot" ...>{{ i }}</div>`. The variable `i` is the loop index (1-based). Each dot shows its number as visible text content. Unless the CSS hides this with `overflow: hidden` or `font-size: 0`, consumers will see numbers inside each dot.
- Evidence: `NyxProgress.vue:50` — `>{{ i }}</div>`
- Fix: Remove `{{ i }}`. If the dot index is needed for debugging, use a `data-index` attribute instead.
- Breaking: Non-breaking.

**M7 — `useKeyboardShortcuts` keyHistory may grow indefinitely if keyup is missed**
`useKeyboardShortcuts.ts` maintains a `Set<string>` of currently-held keys. Keys are added on `keydown` and removed on `keyup`. If the component is focused during a shortcut, then blur or destroy occurs before `keyup` fires, keys remain in the set permanently. Future shortcut matching may then always see a "ghost" modifier held down.
- Evidence: The composable adds a `keyup` listener and a `keydown` listener but has no `blur` listener to clear the history.
- Fix: Add a `window.addEventListener('blur', () => keyHistory.clear())` or clear on component unmount.
- Breaking: Non-breaking.

**M8 — `NyxLoader` belongs in `utils/`, not `classes/`**
`NyxLoader` is a collection of static methods for type-safe data parsing from unknown objects (Firebase document parsing). It has no relationship to UI, Vue, or component rendering. It is a general-purpose data utility. Placing it in `classes/` alongside `NyxLog` (which is genuinely library-internal tooling) misleads consumers about its intended use and suggests it belongs in the UI layer.
- Evidence: `src/classes/NyxLoader.ts` — pure data parsing, zero Vue/DOM imports
- Fix: Move to `src/utils/NyxLoader.ts` (or a `data/` sublayer) and update the `classes/` entry point. Mark as a candidate for its own package.
- Breaking: Changes the import path `nyx-kit/classes` → `nyx-kit/utils`. Breaking for consumers using it.

### Low

**L1 — `CssVariablesDict` type protects nothing**
`type CssVariablesDict = {} | { [key: CssVariableKey]: string | number }`. The `{}` branch makes this equivalent to `unknown` (not `{}` in the intuitive sense — TypeScript's `{}` accepts all non-nullish values). The union collapses to the most permissive type. Every assignment to `CssVariablesDict` is valid. The `CssVariableKey = \`--${string}\`` template literal type is the only value-add here, and it's lost in the union.
- Evidence: `src/types/style.ts:5-6`
- Fix: Remove the `{}` branch. `type CssVariablesDict = Record<CssVariableKey, string | number>`.
- Breaking: Non-breaking (stricter type; no runtime change).

**L2 — `isObject` is in `keydict.ts`, not `object.ts` or `utils.ts`**
`src/utils/keydict.ts` exports both `KeyDict<T>` (a type alias) and `isObject` (a runtime function). These have no thematic relationship. `isObject` should live in a dedicated `object.ts` util or at minimum in `common.ts`. Its current home is a discoverability problem — no one looks in `keydict.ts` for a general object type guard.
- Evidence: `src/utils/keydict.ts:7` — `export const isObject`
- Fix: Move to `src/utils/object.ts` or rename file to `object.ts` and update `index.ts`.
- Breaking: Non-breaking.

**L3 — `useSelectionPosition` is not exported from `src/composables/index.ts`**
`useSelectionPosition.ts` implements selection-based floating positioning (used by NyxEditor's bubble menu). It's not exported from the composables index. Consumers cannot use it. It's either internal-only (fine, but should be documented) or intended for export (in which case, export it).
- Evidence: `src/composables/index.ts` exports only 3 composables; `useSelectionPosition` not among them
- Fix: Either export it with documentation, or add a comment in the file: `// internal — not part of public API`.
- Breaking: Non-breaking.

**L4 — Dead `<dialog>` comment in NyxModal**
`NyxModal.vue:47`: `<!-- <dialog :open="model" ref="elDialog"></dialog> -->` — commented-out code with no explanation of why it was disabled or what blocked the migration to native `<dialog>`. This is a maintenance question mark that will confuse future editors.
- Evidence: `NyxModal.vue:47`
- Fix: Either migrate to `<dialog>` (which provides built-in focus trap and accessibility), or delete the comment and file a docs note explaining the choice.
- Breaking: Non-breaking.

**L5 — `NyxCarousel.isAutoplay` is a pointless computed**
`NyxCarousel.vue:22`: `const isAutoplay = computed(() => props.autoplay)`. This wraps a prop in a computed that adds no transformation. It's used nowhere — the actual autoplay check uses `props.autoplay` directly in `resetInterval()`. This is dead weight.
- Evidence: `NyxCarousel.vue:22`, `NyxCarousel.vue:33` — `resetInterval` reads `props.autoplay` directly
- Fix: Delete `isAutoplay`. It adds confusion without value.
- Breaking: Non-breaking.

## 3. Architectural Assessment

The intended architecture is clean: components are pure Vue SFCs, `useNyxProps` centralises class generation, composables handle reusable logic, types define public contracts, utils are pure functions. The plugin `provide/inject` for global defaults is the right pattern for a configurable library.

The implementation diverges from this intent in two areas:

**1. useNyxProps adoption is inconsistent.** The pattern exists and works well in the ~12 components that use it. But 10 visual components roll their own class logic. The result is a split personality: half the library respects `NyxKit.install()` defaults; half doesn't. There's no clear principle for which components use it and which don't — it appears to be a matter of when each component was written, not a design decision.

**2. NyxLoader is a layer violation.** A component library that includes a Firebase document parser is doing too much. `NyxLoader` has no relationship to Vue, DOM, or rendering. Its presence in `classes/` implies that this library is also an application data layer. That's a scope creep problem that will attract more non-UI code over time.

The teleport positioning system (`useTeleportPositionBase`) is well-designed — SSR-guarded, handles mirroring, driven by CSS variables. The split into Base + derived composable is slightly over-engineered for two consumers (NyxSelect and NyxTooltip), but it's clean.

## 4. API Design Assessment

**Strengths:** Consistent use of `defineModel` for v-model, named emits with typed payloads, slot naming conventions (`tab-${name}`, `tab-button-${name}`) are predictable, `NyxComponentProps` base interface is well-factored.

**Issues:**

- **NyxSelect `type` vs `multiple`** — two props for one concept. See M2.
- **NyxProgress `showValue: boolean | "inside" | "outside"`** — the first argument to `boolean |` makes the type unclear. If `showValue = true`, does the label show inside or outside? The API should be `showValue: boolean` + `valuePlacement: "inside" | "outside"`.
- **NyxTooltip `forceUpdate`** — internal mechanism as public prop. See M3.
- **NyxTable `colWhitelist`/`colBlacklist`** — terminology should be `colInclude`/`colExclude` (acknowledged in previous audit, still unfixed).
- **`NyxSteps` has `themeComplete` and `themeIncomplete` props** — component-specific theming props that don't follow the standard pattern. If every component adds ad-hoc theme props for sub-states, the API surface grows unbounded.
- **NyxTooltip `text` prop is `string | number`** — the `number` branch exists presumably to avoid the consumer casting. Minor, but odd to special-case.

## 5. Styling and Theming Assessment

The token system is excellent. A comprehensive set of CSS custom properties (text, background, theme colours with 5 variants each, spacing, typography, shadows, animation speeds, pixel mode) is well-structured. Using CSS variables means consumers can override tokens globally or per-scope.

**Issues:**

- **Global CSS pollution (see H1)** — the reset is the worst of it. This isn't a preference: shipping a 2011 full-element reset in a published component library CSS will break consumer applications.
- **`NyxProgress variant--${variant}` naming inconsistency (see M1)** — double-dash vs. single-dash in the same library.
- **No documented dark mode strategy** — `variables.css` appears to define separate light/dark token sets, but whether this is driven by `prefers-color-scheme`, a class, or a data attribute is not documented. Consumers can't reliably implement dark mode without knowing this.
- **`flex.css` utility classes** — these are fine for internal use but publish utility classes globally. Conflicts with Tailwind, Bootstrap, or consumer utility CSS are possible.
- **`props.css`** — contents not clear from the filename. Should be named descriptively.
- **NyxTooltip manually builds class string** (`\`theme-${props.theme}\`, \`variant-${props.variant}\`, \`size-${props.size}\``) — bypasses the token fallback chain. If a consumer configures a global theme via the plugin, NyxTooltip ignores it.

## 6. TypeScript Assessment

The type discipline is largely good. Each component has a `.types.ts` file with explicit interfaces. Enums are meaningful and well-named. `defineProps` generics are used throughout.

**Issues:**

- **`useNyxProps` accepts `KeyDict<unknown>`** (`props: KeyDict<unknown>`) instead of something typed as `Partial<NyxComponentProps>`. The composable can receive literally any object. This isn't a runtime problem (it's a trusted internal call), but it makes the function signature misleading and kills IDE guidance for callers.
- **`CssVariablesDict` is effectively `unknown`** (see L1).
- **`NyxKitOptions.defaults` is `Record<NyxKitPrimitive, NyxKitDefaults>`** — the `Record<K, V>` type requires ALL keys of `K` to be present. But `NyxKitDefaults` is used as partial overrides. This should be `Partial<Record<NyxKitPrimitive, NyxKitDefaults>>`.
- **`KeyboardKey` enum has only `Esc`** (see M4) — incomplete and misleading.
- **`NyxProgressProps.showValue: boolean | "inside" | "outside"`** — overlapping type with unclear semantics. `true` is not either string.
- **`NyxSliderProps.tooltip: "always" | "never" | "interact"` (from the template)** — typed as a string union, which is fine, but not exported or aliased as a named type. Minor inconsistency with how other string unions are handled.
- **`NyxKitOptions`/`NyxKitPrimitive` not in `./types`** — consumers can't type-import them correctly (see H5).

## 7. Accessibility Assessment

This is the most significant gap between where the library is and where it needs to be for serious use.

**Good:**
- `NyxSlider`: correct `role="slider"`, `aria-valuenow/min/max`, keyboard support (Arrow keys)
- `NyxSpinner`: `role="progressbar"`, `aria-label`
- `NyxCarousel`: cloned infinite-loop slides have `aria-hidden="true"` and `tabindex="-1"`
- `NyxCheckbox`: `<label for="id">` association
- `NyxModal`: ESC dismissal

**Bad:**
- `NyxModal`: No `role="dialog"`, no `aria-modal`, no `aria-labelledby`, no focus trap. Tab escapes the modal.
- `NyxTabs`: No `role="tablist/tab/tabpanel"`, no `aria-selected`, no `aria-controls`. Arrow key navigation absent.
- `NyxSelect`: No combobox ARIA pattern. Screen readers have no context for what this UI control is.
- `NyxTooltip`: No `aria-describedby` on trigger. Tooltip content is invisible to screen readers.
- `NyxProgress` (line bar): No `role="progressbar"`, `aria-valuenow`, `aria-valuemin`, `aria-valuemax`. (NyxSpinner has it, but NyxProgress doesn't.)

The gap between NyxSlider (fully accessible) and NyxTabs/NyxModal (not accessible at all) suggests components were written at different times with different attention to a11y rather than from a consistent standard.

## 8. Testing Assessment

The previous audit found zero tests. The fix added tests for composables, utilities, and NyxLoader. That's meaningful progress.

**Current state:**
- `useNyxProps.spec.ts` — 36 tests, good coverage, proper edge cases. Best test file in the codebase.
- `useKeyboardShortcuts.spec.ts`, `useKeyPress.spec.ts` — present, basic.
- Utility specs — small and adequate for the simple functions they cover.
- `NyxLoader.spec.ts` — covers the static methods.

**What's still missing:**
- **Zero component render tests.** No `@vue/test-utils` tests for any of the 23 components. Props → DOM output, v-model two-way binding, event emission, slot rendering — all untested.
- **E2E tests are a placeholder.** `e2e/vue.spec.ts` checks for `<h1>You did it!</h1>` — a Vite scaffolding artefact. No E2E coverage of any component behaviour.
- **No test for `useTeleportPositionBase`** — the most complex logic in the codebase, responsible for positioning all floating elements, has no direct tests.
- **No tests for NyxSlider drag logic** — pointer event handling, step snapping, range mode thumb swapping, boundary clamping. The post-audit rewrite is untested.
- **No tests for NyxModal keyboard handling** — ESC dismissal, static mode bypass.

The test suite produces confidence for composables and utilities. It produces zero confidence for component rendering behaviour.

## 9. Build/Package Assessment

**Good:**
- Seven clean entry points with independent sub-path exports
- `sideEffects: ["**/*.css"]` correctly set (fixed in previous audit)
- `vite-plugin-dts` producing type declarations
- Vue externalized from all bundles
- Backward-compat `./compositions` alias maintained

**Issues:**
- **`./types` export doesn't include `NyxKitOptions`/`NyxKitPrimitive`** (see H5)
- **`NyxKitOptions.defaults` typed as `Record` not `Partial<Record>`** — means full object is required
- **`./compositions` backward-compat alias** is present in `package.json` exports but not documented. If it's intentional, document it. If it's a rename typo, remove it.
- **`isObject` lives in `keydict.ts` but is imported by `NyxTable.vue` from `@/utils`** — this works today, but `keydict.ts` is a type-only utility by name. If the file is renamed or reorganised, the import breaks silently.

| Dimension | Score | Justification |
