# Nyx Kit Audit — 20260330-1634

## 1. Executive Summary

Nyx Kit is a Vue 3 component library with decent initial structure but significant architectural drift. The library ships 25+ components, composables, utilities, and a theming system, but suffers from inconsistent patterns, unclear ownership boundaries, and model mutation anti-patterns that will cause pain as the codebase grows.

**What works:**
- Strong TypeScript coverage and component-level testing
- Theming system with theme/size/variant primitives via `useNyxProps`
- Good accessibility baseline (ARIA roles, keyboard navigation in Select)
- Storybook integration for visual testing

**What's annoying:**
- Export chaos: components exported directly, via index.ts, or both—with no clear convention
- NyxTree mutates its model directly, violating Vue 3's reactivity contract and the library's own documented pattern
- Classes folder is a zombie—NyxLoader lives in utils but re-exported from classes with a confusing comment
- Spec files exist for only 6 of 25+ components—documentation is drifting from reality
- NyxSelect has odd UX: sets search input to option *label* on selection rather than value or empty string

**Rot forecast:** Moderate-high. The component surface is large enough that inconsistent patterns will compound. Model mutation in NyxTree is a time bomb—Vue 3 won't detect changes, and consumers will see stale UI.

**Top 3 priorities:**
1. Fix NyxTree model mutation (critical reactivity bug)
2. Standardize component export strategy
3. Complete living spec files for all components

## 3. Findings

### [Critical] NyxTree Mutates Model Directly

**Category:** Maintainability / Rot resistance  
**Time horizon:** Immediate  
**Blast radius:** Local (NyxTree only, but breaks reactivity)

**What's annoying:**  
The component receives `model` via `defineModel` but mutates `node.status` directly on the array items:

```ts
// NyxTree.vue:58-67
function handleSelect(node: NyxTreeNodeBase) {
  if (node.children.length === 0) {
    clearActive(model.value)
    node.status = NyxTreeNodeStatus.Active  // DIRECT MUTATION
  } else {
    node.status = ... // DIRECT MUTATION
  }
  emit('select', node)
}
```

**Why it matters later:**  
Vue 3's `defineModel` only emits `update:modelValue` when the model *reference* changes—not when nested properties are mutated. The component never reassigns `model.value`, so:
1. The emit never fires
2. Vue reactivity won't track the change
3. Consumers using `v-model` won't see updates
4. This contradicts the library's own documented pattern in AGENTS.md (section "Components that mutate nested model data")

**Evidence:**  
- `src/components/NyxTree/NyxTree.vue:51-69` — all mutations are direct on array items
- AGENTS.md explicitly documents the correct pattern: reassign model after mutation

**Suggested fix:**  
Implement the pattern from AGENTS.md:
```ts
function handleSelect(node: NyxTreeNodeBase) {
  const updated = updateNodeInTree(model.value, node.id, (n) => {
    // mutate n.status
  })
  model.value = updated       // reassign to trigger emit
  emit('select', node)
}
```

**Roast line:** NyxTree found the one way to break Vue 3's reactivity that the documentation specifically warns against.

### [High] Export Inconsistency Chaos

**Category:** Consistency / Developer experience  
**Time horizon:** 1-3 months  
**Blast radius:** Cross-cutting (all components)

**What's annoying:**  
Components are exported in three different ways with no pattern:

1. **Direct import from .vue** (majority): `import NyxInput from './NyxInput/NyxInput.vue'`
2. **Via index.ts re-export**: `export { default as NyxTree } from './NyxTree/index'`
3. **Both**: NyxGrid and NyxEditor have index.ts, but `components/index.ts` imports directly from .vue anyway

```ts
// components/index.ts imports directly
import NyxGrid from './NyxGrid/NyxGrid.vue'  // ignores index.ts!
import NyxEditor from './NyxEditor/NyxEditor.vue'  // ignores index.ts!

// But NyxTree uses index.ts
import NyxTree from './NyxTree/index'  // uses index.ts
```

**Why it matters later:**  
- Refactoring becomes risky: changing a component's file structure breaks consumers who import via different paths
- Internal imports are fragile—if someone adds an index.ts that re-exports differently, imports throughout the codebase might resolve differently
- No way to enforce consistency; future components will follow the "whatever works" pattern

**Evidence:**  
- `src/components/index.ts` — imports 24 components, mixes patterns
- `src/components/NyxGrid/index.ts`, `src/components/NyxEditor/index.ts`, `src/components/NyxTree/index.ts` — exist but aren't uniformly used

**Suggested fix:**  
Pick one pattern and enforce it. Recommendation: use index.ts for all multi-file components (sub-components like NyxTreeNode, NyxTableCell, NyxFormField). Import everything through index.ts.

**Roast line:** Three export strategies walk into a library. Nobody knows which one to use at commit time.

### [High] Classes Folder Identity Crisis

**Category:** Architectural integrity  
**Time horizon:** 1-3 months  
**Blast radius:** Local (classes/ and utils/)

**What's annoying:**  
`NyxLoader` lives in `utils/NyxLoader.ts` but is re-exported through `classes/` with a "backward compatibility" comment:

```ts
// src/classes/NyxLoader.ts (2 lines)
// NyxLoader has moved to utils/ — this re-export keeps nyx-kit/classes backward-compatible
export { default } from '../utils/NyxLoader'
```

`NyxLog` lives in `classes/` with no corresponding utility. The split is unclear:
- Classes folder contains: `NyxLoader`, `NyxLog`
- Utils folder contains: `NyxLoader`, plus utilities

**Why it matters later:**  
- Where should new "class-like" utilities go? The boundary is unclear
- The re-export comment acknowledges the move but doesn't explain *why*
- NyxLog has no home in either world—it's a logging utility, not a domain class

**Evidence:**  
- `src/classes/NyxLoader.ts` — re-export comment
- `src/utils/NyxLoader.ts` — actual implementation
- `src/classes/index.ts` — exports both

**Suggested fix:**  
Kill the classes/ re-export or merge NyxLog into utils/. Pick a philosophy: either classes/ contains domain entities and utils/ contains helpers, or everything goes in utils/.

**Roast line:** NyxLoader is playing hide-and-seek with the codebase.

### [Medium] NyxSelect Sets Search Input to Label on Selection

**Category:** Edge-case resilience / UX  
**Time horizon:** 3-6 months  
**Blast radius:** Local (NyxSelect)

**What's annoying:**  
When a user selects an option in single-select mode, the search input is populated with the option's *label* rather than its *value* or cleared:

```ts
// NyxSelect.vue:97-101
} else {
  model.value = value
  searchQuery.value = label  // sets to LABEL, not value
  isOpen.value = false
}
```

**Why it matters later:**  
This creates confusing UX:
- If labels and values differ, the input shows "California" but the model holds "CA"
- Re-opening the dropdown shows the search populated with stale data
- Most select components either show the value or clear the input

**Evidence:**  
`src/components/NyxSelect/NyxSelect.vue:97-101`

**Suggested fix:**  
Set `searchQuery.value = ''` on selection, or use the value. The current label behavior seems unintentional.

**Roast line:** NyxSelect is the only select component that confuses users with what they just selected.

## 4. Systemic Patterns

### Pattern 1: Export Strategy Drift
The codebase has no enforcement mechanism for how components should be exported. Some have index.ts, some don't, some are imported directly. This is a classic "growing pains" pattern where the library evolved without a strict convention.

**Fix:** Add a lint rule or documentation enforcing one pattern.

### Pattern 2: Model Mutation Confusion
Despite AGENTS.md documenting the correct pattern for mutating nested model data, NyxTree doesn't follow it. This suggests either:
- The documentation was written after implementation and never enforced
- The pattern is unclear in practice

**Fix:** Add a test that verifies `update:modelValue` fires when tree nodes are clicked.

### Pattern 3: Zombie Re-exports
The classes/ folder re-exporting from utils/ is a sign of incomplete refactoring. The comment admits it ("backward compatibility") but doesn't clean it up.

**Fix:** Remove the re-export and update consumers to import from the canonical location.

### Pattern 4: Test Surface Coverage
Tests are present but focus on "renders correctly" more than "behaves correctly." The Select tests are the exception—they test keyboard navigation, disabled states, a11y properly.

**Fix:** Expand test coverage to match Select's thoroughness.

## 5. Fix Order

### Fix Now
1. **NyxTree model mutation** — This is actively broken. Vue won't detect changes, consumers will see stale UI. Fix before anyone builds real features on top of it.

### Fix Next
2. **Standardize exports** — Pick index.ts for all multi-file components, import through it consistently. This prevents future drift.
3. **Remove NyxLoader re-export** — Kill the classes/NyxLoader.ts or merge NyxLog into utils/. Pick a boundary.

### Fix Opportunistically
4. **NyxSelect searchQuery** — Small fix, improves UX
5. **Spec files** — Create 1 per sprint until complete
6. **Consistent v-model naming** — Rename to `modelValue` everywhere or document why `model` is preferred

### Monitor Only
- NyxLog hardcoded styles — Low impact, cosmetic
- Large component files — Refactor when touching, not preemptively

---

## 6. What Not to Waste Time On

- **Pixel theme refactoring** — It's isolated, works, and serves a specific aesthetic. Don't force it into the main theming system.
- **Extracting composables from large components preemptively** — Wait until someone needs to reuse the logic
- **Consolidating all tests into one pattern** — Current tests work; improve coverage gradually

---

## 7. Final Verdict

Nyx Kit is a solid Vue 3 component library with good TypeScript foundations and accessibility attention. The component surface is impressive. But it's carrying architectural debt—export inconsistency, model mutation bugs, zombie re-exports, and missing documentation—that will compound as the library grows. The model mutation issue in NyxTree is the most urgent: it's actively broken and will confuse early adopters.

Fix the reactivity bug, standardize exports, and document the component contracts. Do that, and this becomes a maintainable library. Ignore it, and NyxTree will be the haunted component nobody wants to touch in six months.
