# Nyx-kit Audit — 20260315-1949

---

## 1. Executive Summary

Nyx-kit is a well-intentioned Vue 3 component library with a coherent design system, consistent API patterns, and good architectural instincts. The composition API is used throughout correctly, the type system is meaningful, and the design token approach is sound.

**Main strengths:** consistent prop/emit patterns, a well-structured CSS variable system, smart teleport positioning, strong TypeScript conventions, pixel mode as a first-class feature.

**Main risks:** zero unit test coverage, several real template bugs in shipped components, no `sideEffects: false` in package.json (hurts tree-shaking for CSS consumers), SSR incompatibility baked into multiple composables, and a few API design gaps that will become breaking changes as the library matures.

**Ready for wider adoption?** Not yet. The bugs in NyxInput, NyxTextarea, NyxModal, and NyxBreadcrumbs need to be fixed first, and the absence of any unit tests means regressions will go undetected. With those addressed, the library is architecturally solid enough for internal team adoption. Public/open-source adoption requires test coverage and a few accessibility fixes.

## 2. Severity Overview

### Critical

**C1 — NyxInput and NyxTextarea: `id` variable is referenced but never declared**
- Severity: Critical
- Why it matters: The `id` attribute on `<input>` and `<textarea>` is used to associate the `<label>` element via `for`. If `id` is undefined, the label association is broken for every consumer using these components in a form. This is both a functional bug and an accessibility regression.
- Evidence: `NyxInput.vue` and `NyxTextarea.vue` reference `:id="id"` in the template but `id` is never declared as a prop, computed, or ref in `<script setup>`.
- Fix: Add an `id` prop (optional string) with a generated fallback using `generateRandomString()`, matching the pattern already used in `NyxCheckbox` and `NyxFormField`.
- Breaking: Non-breaking (adding optional prop).

**C2 — NyxModal: template references undeclared variables**
- Severity: Critical
- Why it matters: `customClass`, `title`, and `confirmText` are referenced in the template without `props.` prefix, which in `<script setup>` means they must be declared as top-level variables. If they are absent, the template will throw at runtime or silently render nothing.
- Evidence: `NyxModal.vue` template uses `customClass`, `:title`, and `confirmText` directly.
- Fix: Prefix with `props.` or destructure explicitly from `props` with `toRefs`.
- Breaking: Non-breaking (bug fix).

### High

**H1 — No unit tests whatsoever**
- Severity: High
- Why it matters: There are no `.spec.ts` files anywhere in the codebase. Every refactor, every composable change, every utility edit is unguarded. The bugs in C1 and C2 are symptoms of this — they would have been caught by even a basic render test.
- Evidence: Vitest is configured and listed in scripts, but `src/**/*.spec.ts` returns no matches.
- Fix: Start with composables (`useNyxProps`, `useTeleportPosition`, `useKeyboardShortcuts`) and utilities (`number.ts`, `NyxLoader`). These are pure or near-pure and easy to test. Then add component smoke tests using `@vue/test-utils`.
- Breaking: Non-breaking.

**H2 — No `sideEffects` field in package.json**
- Severity: High
- Why it matters: Without `"sideEffects": ["**/*.css", "**/*.scss"]` (or `false` for JS), bundlers cannot safely tree-shake JS modules that import CSS. Consumers using webpack or Rollup will either over-include modules or get broken styles.
- Evidence: `package.json` has no `sideEffects` field.
- Fix: Add `"sideEffects": ["**/*.css"]` to package.json.
- Breaking: Non-breaking.

**H3 — SSR incompatibility in composables and components**
- Severity: High (for any consumer targeting SSR/Nuxt/Vite SSR)
- Why it matters: `useTeleportPosition` accesses `window` (for resize/scroll listeners and rem-to-px conversion) and `document` without guards. `vClickOutside` accesses `ownerDocument`. Several components use `<Teleport to="body">` directly. None of these are guarded with `if (typeof window !== 'undefined')`.
- Evidence: `useTeleportPosition.ts`, `vClickOutside.ts`, multiple component templates.
- Fix: Guard window/document access. Use `onMounted` for listener setup (already done in most cases), but the rem conversion and initial position calculation also need guarding.
- Breaking: Non-breaking.

**H4 — NyxBreadcrumbs: invalid attributes on `<a>` elements**
- Severity: High
- Why it matters: `:size` and `:variant` are bound as attributes on native `<a>` tags. These are not valid HTML attributes and will appear as unrecognised attributes in the DOM, producing console warnings and potentially causing issues with certain attribute-based CSS selectors or testing frameworks.
- Evidence: `NyxBreadcrumbs.vue` template, anchor element bindings.
- Fix: Remove these bindings from the `<a>` tag and apply size/variant via CSS classes instead.
- Breaking: Non-breaking.

### Medium

**M1 — `gradient` and `backlight` are not type-guarded against non-Solid variants**
- Severity: Medium
- Why it matters: `useNyxProps` emits a console warning when `gradient` or `backlight` is used with a non-Solid variant, but there is no TypeScript-level enforcement. Consumers get a runtime warning instead of a compile error. The warning is also swallowed silently in production if the consumer suppresses logs.
- Evidence: `useNyxProps.ts` runtime check; no type-level constraint.
- Fix: Consider a discriminated union props type, or at minimum document this constraint clearly in the Storybook story and `component-model.md`.
- Breaking: Non-breaking.

**M2 — `NyxTree` is not exported from `src/components/index.ts`**
- Severity: Medium
- Why it matters: Consumers importing from `nyx-kit/components` cannot access `NyxTree`. It must be imported directly from the full `nyx-kit` bundle, which undermines the selective-import model.
- Evidence: `src/components/index.ts` does not include `NyxTree`.
- Fix: Add `export { default as NyxTree } from './NyxTree/index'` to `src/components/index.ts`.
- Breaking: Non-breaking.

**M3 — `NyxLoader` is architecturally misplaced**
- Severity: Medium
- Why it matters: `NyxLoader` is a data-parsing utility, not a UI class. It has no relationship to the component library and handles Firebase timestamp formats — a very specific concern that has no place in a generic UI kit. It adds dead weight to the `classes` bundle for any consumer who never needs it, and its presence signals unclear library boundaries.
- Evidence: `src/classes/NyxLoader.ts`.
- Fix: Either move it to `utils/` and export it from there, or remove it from nyx-kit entirely if it belongs to a separate package.
- Breaking: Moving it changes the import path (breaking for existing consumers of `nyx-kit/classes`). Can be mitigated with a re-export alias.

**M4 — `NyxSelect` search input uses a string ref key inconsistently**
- Severity: Medium
- Why it matters: `useTemplateRef('nyx-select-input')` uses a string key that must match `ref="nyx-select-input"` in the template exactly. If either side drifts, the ref silently becomes `null` at runtime with no TypeScript error.
- Evidence: `NyxSelect.vue` lines 24 and 119.
- Fix: Use `const elInput = useTemplateRef<HTMLInputElement>('elInput')` with a simple, consistent name. Or use a regular `ref` and assign it via template ref.
- Breaking: Non-breaking.

**M5 — `NyxCarousel` infinite loop duplicates slide nodes in the DOM**
- Severity: Medium
- Why it matters: Cloned slides are rendered as actual DOM nodes. If slides contain interactive content (buttons, links, inputs), their duplicates are also focusable and reachable by screen readers, producing duplicate interactive elements that are invisible to sighted users but navigable via keyboard/AT.
- Evidence: `NyxCarousel.vue` — slide duplication for infinite loop.
- Fix: Add `aria-hidden="true"` and `tabindex="-1"` to all cloned slides. Alternatively, use CSS-based infinite looping to avoid DOM duplication.
- Breaking: Non-breaking.

**M6 — `NyxLog.getCaller()` is dead code**
- Severity: Medium (low impact, but signals incomplete work)
- Why it matters: The method is defined but never called anywhere in the codebase. Dead code in a published library increases bundle size and creates confusion about intent.
- Evidence: `NyxLog.ts` lines 37–45.
- Fix: Remove it.
- Breaking: Technically breaking if a consumer somehow calls `NyxLog.getCaller()` directly — extremely unlikely.

**M7 — `NyxSlider` mouse drag does not use pointer capture**
- Severity: Medium
- Why it matters: The slider handles `mousemove` on `document` during drag, but does not call `el.setPointerCapture()`. On some browsers, fast mouse movement can escape the element and lose the drag state, or conflict with other event listeners on the document.
- Evidence: `NyxSlider.vue` drag event handling.
- Fix: Use `pointerdown`/`pointermove`/`pointerup` with `el.setPointerCapture(event.pointerId)` instead of mousedown/mousemove on document.
- Breaking: Non-breaking.

### Low

**L1 — ESLint config in `package.json` has quadruplicated Storybook extend entries**
- Severity: Low
- Evidence: `package.json` `eslintConfig.extends` array has `"plugin:storybook/recommended"` four times.
- Fix: Deduplicate to one entry.

**L2 — `src/stores/` directory exists but is empty / unexplored**
- Severity: Low
- Why it matters: An empty `stores/` directory suggests either planned Pinia usage or leftover scaffolding. Either way it should be removed or documented.

**L3 — `NyxSpinner` uses an SVG with no `aria-label` or `role`**
- Severity: Low
- Why it matters: The spinner conveys a loading state but has no accessible label. Screen readers will either ignore it or announce the raw SVG content.
- Fix: Add `role="status"` and `aria-label="Loading"` (or make the label a prop).

**L4 — CSS `additionalData` in `vite.config.ts` uses `@use` on a `.css` file**
- Severity: Low
- Why it matters: `@use "@/styles/index.css"` is injected into every SCSS file via `additionalData`. Sass `@use` is designed for `.scss`/`.sass` modules — using it on a plain `.css` file is technically valid but non-idiomatic and may produce unexpected results in future Sass versions.
- Fix: If global SCSS variables/mixins need to be injected, extract them to a `_globals.scss` partial and use that.

**L5 — `NyxFormField` provides an `id` to slotted children but slot consumers must know to inject it**
- Severity: Low
- Why it matters: The `id` is provided via slot props, but NyxInput doesn't accept an `id` prop yet (see C1), so the pairing is broken. Even after C1 is fixed, the pattern requires consumers to manually thread the id from slot props into the child component, which is fragile and non-obvious.
- Fix: After fixing C1, verify NyxFormField's slot pattern works end-to-end and document it.

## 3. Architectural Assessment

The library has good instincts. The `provide/inject` for global options, the single `useNyxProps` pipeline, and the seven-bundle split are all correct for a component library.

**Library boundaries are mostly respected** — components don't reach into each other, and shared logic lives in composables. The main violation is `NyxLoader`, which has nothing to do with UI and imports library-specific concerns (Firebase timestamps).

**Primitives vs composites** are blurred in a few places. `NyxCard` is simultaneously a layout primitive and a content composite (it knows about `NyxMedia` internally). If consumers need a card without media, the coupling is invisible but present. Consider whether the media slot could be handled purely via slots rather than an internal `NyxMedia` render.

**Scalability concern:** `useNyxProps` is the single chokepoint for all visual prop logic. As more components need custom class generation logic (NyxTabs has `floating`, NyxProgress has `showValue` placement, NyxSlider has `direction`), each adds ad-hoc handling outside `useNyxProps` rather than through it. This will fragment over time.

**The `pixel` mode** is architecturally sound as a first-class feature but creates a hidden constraint: it silently does nothing on non-Solid variants. This should be documented prominently.

## 4. API Design Assessment

**Consistency is good.** `theme`, `size`, `variant`, `shape`, `disabled` appear in the same form across components. `defineModel` is used correctly. Emits are named as past-tense-free verbs (`close`, `confirm`, `cancel`).

**Inconsistencies to address:**
- `NyxTabs` requires `tabs: string[]` as a required prop, but `NyxSteps` accepts either `number` or `string[]`. The different approaches are justified by different semantics but the divergence may surprise consumers.
- `NyxTable` uses `colWhitelist`/`colBlacklist` — "whitelist/blacklist" terminology should be replaced with `include`/`exclude` or `allow`/`deny` for inclusive language and clearer semantics.
- `NyxProgress` prop `showValue` accepts `boolean | "inside" | "outside"`. This mixed type is awkward — consider separating into `showValue: boolean` and `valuePosition: "inside" | "outside"`.
- `NyxButton` emits `@click` but several other components (NyxBadge, NyxCard) also emit `@click`. This is fine, but there is no shared interface for "clickable" components.

**Controlled vs uncontrolled:** All components are fully controlled (v-model required or optional). There is no uncontrolled mode. For a component library, this is fine and simpler to reason about.

**Composability:** Components are not easily composable with each other. For example, there is no `NyxField` that wraps an input with a label, error message, and helper text as a single unit. `NyxFormField` exists but only handles the label. Form error and helper text are missing.

## 5. Styling and Theming Assessment

**Token strategy is strong.** The CSS custom property system is comprehensive, semantically named, and covers all necessary dimensions. The five-variant per-theme structure (base, light, dark, highlight, alt) is well-considered.

**Override ergonomics for consumers are poor.** There is no documented way for a consumer app to extend or override the theme tokens at the app level. The library sets all tokens at `:root`, but there is no guidance on how to scope a custom theme (e.g. `.my-app { --nyx-c-primary: hotpink; }`). This should be documented.

**Dark mode:** The token file defines dark mode tokens but it is not clear whether the library ships a dark mode switcher or expects the consumer to toggle a class/attribute. This should be explicit in `design-system.md`.

**Style leakage risk:** `src/styles/flex.css` and `src/styles/base.css` set global styles on element selectors (`*`, `body`, `html`, `a`, `button`, etc.). These are included in `style.css` and will affect the consumer's global styles if they import the CSS. This is a significant risk for micro-frontend or mixed-component-library scenarios.

**The `additionalData` SCSS injection** means that if a consumer installs nyx-kit and uses Vite with their own SCSS, the nyx-kit global styles will not automatically inject — they must import `nyx-kit/style.css` separately. This is correct but should be documented clearly.

## 6. TypeScript Assessment

**Generally good.** Every component has a `.types.ts` file, enums are used for fixed sets of values, and generic components (`NyxTable<T>`, `NyxCarousel<T>`) are correctly typed.

**Issues:**
- `useNyxProps` takes `props: any` (or a broad type) — the actual constraint should be typed against `Partial<NyxComponentProps>` to prevent passing arbitrary objects.
- `CssVariablesDict` is typed as `{} | { [key: CssVariableKey]: string | number }`, which is redundant — `{}` is a supertype of the second. It should just be `{ [key: CssVariableKey]: string | number }`.
- `NyxLoader.loadEnum` uses a generic `T` with no constraint — the enum value check uses `Object.values(enumObj)` which works at runtime but has no TypeScript guarantee that `T` is actually an enum type.
- `KeyboardKey` enum contains only `Esc = 'Escape'`. This is fine, but consumers who want to handle other keys cannot use `useKeyPress` with other keys without modifying the library enum. Consider accepting a raw `string` alternative.
- The `NyxKit` plugin's `NyxKitOptions` type is exported from `src/main.ts` but not from `src/index.ts` or `src/types/`. Consumers who want to type their plugin options must import from an unexpected path.

## 7. Accessibility Assessment

**Reasonable baseline, but gaps exist.**

**Good:**
- NyxSlider has correct `aria-valuenow`, `aria-valuemin`, `aria-valuemax`, `role="slider"`.
- NyxCheckbox links label to input via `for`/`id`.
- NyxModal uses `<Teleport>` to body, avoiding stacking context issues.
- NyxBreadcrumbs uses `<nav>`.
- NyxMedia uses `<figure>` and `<figcaption>`.

**Missing or broken:**
- NyxInput and NyxTextarea `id` bug (C1) means label association is broken everywhere.
- NyxModal has no `role="dialog"`, `aria-modal="true"`, or `aria-labelledby`. Without these, screen readers do not announce it as a modal and do not trap focus.
- NyxModal does not trap focus. When open, Tab will exit the modal and interact with background content.
- NyxSpinner has no `role="status"` or `aria-label` (L3).
- NyxCarousel cloned slides are focusable and visible to AT (M5).
- NyxSelect has a hidden native `<select>` for accessibility, but it is unclear whether it is properly linked to the visible trigger (no matching `id`/`aria-labelledby` visible in the audit).
- NyxTooltip: the trigger element has no `aria-describedby` linking it to the tooltip content. Screen readers will not announce the tooltip.
- NyxTabs: no `role="tablist"`, `role="tab"`, `role="tabpanel"`, or `aria-selected` on tab buttons.

The accessibility gaps in NyxModal, NyxTabs, NyxTooltip, and NyxSelect are significant. These are high-interaction components and the missing ARIA patterns are not optional niceties.

## 8. Testing Assessment

**There are no unit tests.** The `vitest` config exists, `@vue/test-utils` is installed, but there are zero `.spec.ts` files in the entire `src/` tree.

E2E tests exist in `e2e/` (Playwright), but their content and coverage were not reviewed in this audit.

Storybook stories exist for all components, which provides some manual visual coverage, but stories are not a substitute for automated tests.

**What is not covered at all:**
- `useNyxProps` class generation logic
- `useTeleportPosition` position calculations and mirroring
- `useKeyboardShortcuts` modifier key handling
- All utilities (`number.ts`, `string.ts`, `array.ts`, `url.ts`)
- `NyxLoader` parsing paths, including error cases
- Component-level render tests (props, slots, emits, v-model)
- Keyboard interactions in NyxSlider, NyxModal, NyxSelect

**Risk:** the two critical template bugs (C1, C2) have existed undetected precisely because there are no render tests.

## 9. Build / Package Assessment

**Strengths:**
- Seven independent bundles enabling selective imports.
- Vue correctly externalized.
- `vite-plugin-dts` with `rollupTypes: true` produces clean declaration files.
- Backward-compat `./compositions` alias pointing to the renamed `composables` bundle is handled correctly.

**Issues:**
- No `"sideEffects"` field (H2). This is the most impactful omission.
- The `./types` export re-exports runtime enums. This means `nyx-kit/types` is not a types-only import — it ships runtime JavaScript. Consumers who do `import type { NyxTheme } from 'nyx-kit/types'` will not get the enum values at runtime. Consider whether the types bundle should also be the runtime enum bundle (and rename it accordingly), or separate runtime enums from type-only exports.
- `NyxKitOptions` is not exported from `src/types/index.ts`, forcing consumers to import it from `src/main.ts`.
- `tsconfig.json` at the root and `tsconfig.app.json` both exist — verify there is no duplication in what gets included in the published types.

## 11. Refactor Roadmap

### Phase 1 — Safe quick wins (no API changes)
- Fix C1 (NyxInput/NyxTextarea `id`)
- Fix C2 (NyxModal template variables)
- Fix H4 (NyxBreadcrumbs invalid attributes)
- Fix H2 (`sideEffects` in package.json)
- Fix M2 (export NyxTree from components index)
- Fix M6 (remove NyxLog.getCaller dead code)
- Fix L1 (deduplicate ESLint config)
- Fix L3 (NyxSpinner aria-label)
- Add `NyxKitOptions` to `src/types/index.ts` exports
- Add unit tests for all utilities and composables

### Phase 2 — Structural improvements (no breaking API changes)
- Add ARIA to NyxModal (dialog role, focus trap, labelledby)
- Add ARIA to NyxTabs (tablist/tab/tabpanel/aria-selected)
- Add `aria-describedby` to NyxTooltip
- Fix NyxCarousel cloned slide accessibility (aria-hidden on clones)
- Guard all `window`/`document` access for SSR safety
- Replace `colWhitelist`/`colBlacklist` with `colInclude`/`colExclude`
- Migrate NyxSlider drag to Pointer Events API
- Move or remove NyxLoader from the library
- Add E2E tests for keyboard interactions (Modal, Select, Slider, Tabs)

### Phase 3 — Breaking changes / next major version
- Rename `./compositions` export to `./composables` only (drop backward compat alias)
- Refactor `NyxProgress` `showValue` prop (split into `showValue: boolean` + `valuePosition`)
- Add `NyxFormField` that composes label + input + error + helper text as a unit
- Separate runtime enum exports from type-only exports in `./types` bundle
- Consider form validation composable (`useNyxForm`)
- Consider `NyxField` as a first-class form composition primitive
